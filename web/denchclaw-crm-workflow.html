How a stranger becomes revenue, one legal move at a time.

The DenchClaw CRM is the system of record for the whole Tantra engine suite. Four automation engines — outreach, nurturing, content, personalization — do the talking; the CRM remembers who every contact is, what happened to them, where they stand in two pipelines, and which engine should work them next. This page walks the complete machine, end to end, with nothing skipped: every stage, every transition, every automation, every table.

The Big Picture

Five callers, one API, one database. The engines and the human dashboard never touch PostgreSQL directly — everything flows through the CRM's HTTP API, which enforces authentication, tenant isolation, and the pipeline state machines on every single request.

The one-sentence version Engines generate attention and conversations; the CRM turns those into an auditable ledger of contacts moving through two pipelines — and refuses any move the pipeline rules don't allow.
2

A Contact Is Born

Contacts enter three ways: an engine scrapes or imports a prospect list (POST /contacts/bulk-import), an engine or the dashboard creates one directly (POST /contacts), or the Tantra chat creates one on your behalf. Every new contact starts at marketing stage sourced with a dedicated UUID contact id — the identity every other table hangs off.

Identity
id UUID — referenced by activity, deals, conversations, handoffs, events
Birth state
marketing_stage = 'sourced' · legacy deal_stage = 'lead'
What's stored
name, email, phone, company, title, linkedin_url, source, tags[], UTM source/medium/campaign/content, metadata JSON, lead score
First activity
a created / prospect_loaded row lands in the activity feed immediately
Attribution
every CTA link the engines send is UTM-tagged; those UTMs are saved on the contact for source attribution
Tenancy
company_id stamped from the caller's header — rows never leak across tenants

The Marketing Pipeline

Nine stages that track a contact from raw scrape to marketing-qualified lead. The map below is not decoration — it is the literal state machine stored as JSONB in crm_pipeline_configs (seeded by migration 003, corrected by 006). The server checks every requested move against it and answers 409 + the allowed list for anything illegal. An arrow you don't see here is a move that cannot happen.

4

Engagement & the Unified AI Inbox

Between stage moves, everything a contact does is logged. Two systems capture it: the activity feed (an append-only diary per contact) and the Unified AI Inbox (real two-way conversations per channel). Both feed the lead score.

The activity feed — contact_activity Every event an engine reports lands here as POST /contacts/:id/activity:
  • Message sends on every channel, with the channel recorded — the doc requirement that "any activities on any channels are logged in the contact activity feed."
  • Engagement events that bump the numeric lead score: email_opened +2 · email_clicked +5 · email_replied +10 · whatsapp_read +3 · whatsapp_replied +10 · sms_replied +8 · linkedin_connection_accepted +5 · linkedin_message_replied +10 · video_watched +15 · video_completed +20 · call_booked +25 · call_completed +30 · form_submitted +15 · registered +15 · cta_clicked +10 · proposal_viewed +15 · payment +50.
  • Stage changes (written automatically by every advance, with from/to/reason/actor).
  • Personalization context — the research the personalization engine produces per prospect.
  • Notes from humans via the dashboard modal or Tantra chat.
Score crosses 80 → label flips to hot; a cold contact crossing 50 → warm. Hot/warm contacts surface in the follow-ups queue when they go quiet.
The inbox — conversations + messages One open conversation per contact per channel, enforced by a unique index. Find-or-create is idempotent:
  • POST /conversations {contact_id, channel} returns the existing open thread if there is one — engines can call it blind.
  • Messages carry direction, channel, body, ai_generated, intent, and provider_message_id.
  • Webhook dedupe: the same provider message delivered twice (Resend/Twilio love retries) yields exactly one row — unique on (company, provider_message_id).
  • Escalation: conversations have assignee: ai | human. The ~10% the AI can't handle get PATCH {assignee:"human"} and appear in the dashboard's "Needs Human" queue; a human can hand back to AI.
5

The MQL → Sales Handoff

When marketing produces an MQL, ownership must transfer to Sales exactly once — even with several engine workers racing. The prospect_inbox table is that baton: a CRM-owned queue with atomic claiming. It also serves engine→engine handoffs (e.g. outreach → nurturing).

The Sales Pipeline

Sales lives on a separate object: the deals row created by the handoff claim. That's deliberate — a contact can sit in marketing nurture and have an open deal at the same time without the two states fighting. Twelve stages, seeded by migration 003 and corrected by 006 + 007. Same enforcement: illegal move → 409 + allowed.

Custom pipelines (migration 009) The Pipelines tab can define extra deal pipelines (any key besides marketing/sales). Deals carry an optional pipeline_key; NULL means the built-in sales machine above. Custom-pipeline deals move freely (the operator designed them) and are deliberately excluded from the sales state machine, the sales funnel, the handoff auto-deal check, and the nurture recycle — so bespoke boards can't corrupt the revenue numbers.

Analytics — Events to Funnels

Engines fire campaign events at the CRM; the CRM double-writes them — a raw forensic log plus a pre-aggregated rollup — so dashboards never scan millions of rows. Nine event types, one per meaningful campaign moment.

8

The Dashboard & the Tantra Assistant

The human window into everything above: a single-page dashboard at staging.usetantra.com/crm/ behind one fixed username/password. nginx checks the login, then injects the internal API key server-side — the browser never holds a credential.

Dashboard
stat cards (contacts, new-this-week, open deals, won value, 30-day sends), marketing pipeline overview, contacts table — every row opens the detail modal
Deals board
drag-and-drop kanban for the sales pipeline, the marketing pipeline, and a companies rollup. Drops call the real advance APIs — an illegal drop bounces back with the allowed-moves toast
Contacts
searchable, filterable list (score, source) with create / edit / delete and bulk actions (tag, delete). Stage is deliberately NOT bulk-editable — the state machine owns it
Companies
account records with contact + deal rollups (counts and value); auto-identified from any ingest — a company row is created + linked (company_ref_id) once 2+ contacts share an employer name, and a deal force-creates it. Click a row for the account's contacts + deal pipeline
Pipelines
edit the built-in stage machines (colors, labels, order) or create custom deal pipelines — stored back into crm_pipeline_configs
Inbox
the conversation queue with Open / Needs-Human / All / Closed filters, escalate, hand-back-to-AI, close
Analytics
30-day cards, both funnels in pipeline order, by-channel and by-campaign tables — live from the rollups
Contact modal
every field on the record (identity, stages, score, tags, UTMs, follow-ups, contact id) + the full activity feed + add-note — the human-readable face of the ledger
Tantra — the chat assistant A natural-language operator sitting on the Dashboard. Each turn it receives a live snapshot (stats + up to 80 compact contacts) and either answers in prose or emits a strict JSON action block the server executes through the same public routes — so every rule on this page still applies. Actions: find_contacts, create_contact, update_contact, add_note, delete_contact. Guard-rails: no invented data (it must ask for missing details), duplicate emails refused with an offer to update, each write applied at most once per turn, and deletion requires you to literally type DELETE — the server checks your message, not the model's claim. Runs on Cloudflare Workers AI (Llama 3.3 70B).

The Data Model — All Ten Tables

Everything the machine knows lives in ten tables inside the isolated denchclaw database. Every table carries company_id — the tenant stamp — and every query filters on it. The CRM now runs as a single tenant, tantra (migration 011 folded the legacy growthclub/dev_company ids); the account entity above is the separate companies table, not the tenant.

Security & Multi-Tenancy

Four gates, in order, on every request. Two are about who you are, two are about what you may touch. A different HTTP status marks each failure so callers can tell them apart.

11

Operations — How It Stays Alive

Boring on purpose. The service is built to survive database pressure, restart cleanly, and never mutate its own schema.

Process
pm2 app denchclaw-crm, port 3100 loopback. Deploy = git pull + pm2 restart (never reload — it silently fails on fork-mode apps)
Startup resilience
listens first, probes the DB after; /health answers 503 while retrying with exponential backoff + jitter (1s→2s→4s… cap 30s) forever. Only a config error (missing DATABASE_URL) exits
Connection budget
shared box Postgres, max_connections=200; CRM pool capped at DB_POOL_MAX=10; box-wide rule: all pools + workers < 150
Schema changes
numbered files in migrations/, applied by a human operator with a pg_dump backup first — the app never auto-migrates, agents never run DDL on the live DB
Tests
npm test boots a throwaway Docker postgres:16, applies every migration from zero, and runs the 46-case contract harness — never against staging
Drift protection
GET /pipeline/transitions is byte-pinned by automation_core's CI drift test — the engines' mirror and the CRM must agree or CI fails
UI hosting
nginx serves /var/www/crm/ at /crm/ with Basic auth and injects the API headers; this page lives there too
